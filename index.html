Logstash: The Cost of Every Event

From simple pipelines to production bottlenecks

Pratik

Open with a show of hands: who has tried to fix a slow Logstash pipeline by increasing heap, workers, or batch size first? Set the frame: this talk is not anti-Logstash. It is about treating every filter as code that runs for every event.

Delayed Logs

Logs are delayed by 10 minutes. What knob do you turn first?
  • Increase Logstash heap?
  • Add pipeline workers?
  • Increase batch size?
  • Blame Elasticsearch?

Usually: measure the pipeline before turning knobs.

Let the room vote before revealing the last line. The point is not that these knobs are bad. The point is that they are second-order until we know where time is being spent. Slow pipelines usually come from filter cost, output pressure, queue behavior, or schema/indexing effects downstream.

Working Principle

Logstash filters are production compute, not glue code.

Usually cheaper

Normalize at source, parse repeated shapes, keep mutations shallow.

Cost grows quickly

Regex-heavy parsing, Ruby filters, high-cardinality field names, external lookups.

Emphasize that Logstash is excellent at controlled ingestion and transformation. The risk is putting unbounded or hard-to-predict work into the hot path and then scaling traffic into it. Dynamic field names are especially dangerous downstream because they can create mapping/cardinality pressure even when Logstash itself is still moving events.

Mental Model

input {
  # collect events
}

filter {
  # transform events
  # most production bugs live here
}

output {
  # send events somewhere
}

Inputs collect. Outputs send. Filters spend CPU.

Keep the model simple: inputs decide what becomes an event, filters spend CPU per event, outputs introduce downstream coupling. Most examples in the talk intentionally isolate filters so parser cost is easier to see.

Where does your pipeline sit?

Invite people to click a rung and inspect the sample configuration. Stress that this is a decision ladder, not a law of physics. A small regex on small volume is fine; repeated expensive work at high volume is the problem.

  }
}

output { stdout { codec => rubydebug } }
./commands/01-hello.sh
Run the command and point at the original line under the `message` field. Explain that filters do not parse anything unless we explicitly tell them how.

Stable Log Shape

2026-05-24T10:15:30Z api-01 INFO user=pratik action=login status=200 latency_ms=34
Do we need regex for this?

No, if the format is deterministic.

This line has fixed separators and named key-value fragments. That makes it a good candidate for Dissect or a source-side structured format.

Demo 2A: Parse with Dissect

filter {
  dissect {
    mapping => {
      "message" => "%{log_ts} %{service} %{level} user=%{user} action=%{action} status=%{status} latency_ms=%{latency_ms}"
    }
  }

  mutate {
    convert => {
      "status" => "integer"
      "latency_ms" => "integer"
    }
  }
}
./commands/02-dissect.sh
Run the command and point out that `status` and `latency_ms` are converted after extraction. Mention the key tradeoff: Dissect is fast for reliably repeated text and brittle when the shape varies.

Demo 2B: Parse with Grok

filter {
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:log_ts} %{DATA:service} %{LOGLEVEL:level} user=%{DATA:user} action=%{DATA:action} status=%{NUMBER:status:int} latency_ms=%{NUMBER:latency_ms:int}"
    }
  }
}
./commands/03-grok.sh

Both work. Which one should sit on the hot path?

Grok is the right tool when log shape varies or you need existing pattern libraries. The point is to choose it deliberately, test it with representative volume, and avoid broad patterns where narrow ones would work.

Benchmark Gotchas

  • sincedb can make Logstash process zero events.
  • Use separate --path.data for each run.
  • Avoid rubydebug for benchmarks. Console output dominates.
  • Elasticsearch output measures parser + network + bulk indexing.
  • JVM warmup makes first runs noisy.
The main correction here is to avoid false confidence. A benchmark can accidentally measure stdout, disk, network, sincedb state, or Elasticsearch indexing instead of the parser you meant to compare.

Demo 3: Parse Failures

bad log line with no structure
if "_grokparsefailure" in [tags] {
  mutate { add_field => { "parse_status" => "failed" } }
} else {
  mutate { add_field => { "parse_status" => "ok" } }
}
./commands/06-failure-tags.sh

A parser failure is manageable. An unobserved parser failure is data loss in slow motion.

Grok adds `_grokparsefailure` by default when no pattern matches. The operational move is to route, count, and sample failures instead of letting malformed events blend into normal data.

Type Mismatch Check

mutate {
  convert => { "status" => "integer" }
}

if [status] == "200" {
  mutate { add_field => { "success" => true } }
}
./commands/07-type-bug.sh

After conversion, compare status as a number.

Let the audience read it first. The bug is small but common: conditions after `mutate { convert => ... }` should match the converted type.

Backpressure Signals

queue.type: persisted
path.queue: /var/lib/logstash/queue
queue.max_bytes: 1gb
./commands/08-node-stats.sh
  • events.in, events.filtered, events.out
  • duration_in_millis
  • queue_push_duration_in_millis

Persistent queues buffer pressure. When they fill, inputs are back-pressured.

Persistent queues are useful for resiliency and temporary downstream stalls, but they are bounded. If the queue keeps growing, the system is spending stored capacity. Once the limit is reached, Logstash applies back pressure to inputs that support it.

Hot Path Review

grok {
  match => {
    "message" => "%{GREEDYDATA:raw_prefix} user=%{DATA:user} action=%{DATA:action} payload=%{GREEDYDATA:payload}"
  }
}

ruby {
  code => 'event.set("payload_length", event.get("payload").length)'
}

mutate {
  add_field => { "tenant_%{user}" => "active" }
}
./commands/09-final-debugging.sh

Review cost, cardinality, and failure behavior before this ships.

Walk through three review points: `GREEDYDATA` at the front can do more work than needed, Ruby code must be safe and cheap under concurrency, and dynamic field names like `tenant_%{user}` can create mapping/cardinality pressure downstream. Ask what should be moved to the source, narrowed, or quarantined.

Operating Checklist

  • Can you name the most expensive filter?
  • Can malformed events be quarantined?
  • Can you measure events/sec per pipeline?
  • Can you explain queue growth?
  • Can your schema survive arbitrary input?
Do not ask "Can Logstash do this?" Ask "Should this run on every event?"
Close by shifting the habit from capability to cost. The safest Logstash pipelines are explicit about parser choice, failure routing, throughput measurement, queue behavior, and schema boundaries.
