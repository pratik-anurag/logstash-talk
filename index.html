Logstash - As we know it

From simple pipelines to production gotchas

Pratik

Open with: Raise your hand if you have ever fixed a slow logging pipeline by increasing heap or workers first.

The Mystery

Logs are delayed by 10 minutes. What knob do you turn first?
  • Increase Logstash heap?
  • Add pipeline workers?
  • Increase batch size?
  • Blame Elasticsearch?

Usually: measure the pipeline before turning knobs.

Let audience vote. Then say most slowdowns are quiet architectural decisions, not one dramatic bug. Performance loss usually comes from a few overlooked choices: too much work in Logstash filters, bad ingestion patterns, invisible backpressure, and schema/index issues downstream.

Today's Rule

Logstash filters are production compute, not glue code.

Cheap-ish

Normalize at source, deterministic parsing, shallow mutations.

Expensive fast

Regex everywhere, Ruby filters, dynamic fields, external lookups.

Logstash as a "production compute" or "compute engine" means leveraging its processing capabilities to ingest, parse, enrich, and transform massive streams of raw data in live, production environments before delivering them to downstream storage or analytics platforms

Mental Model

input {
  # collect events
}

filter {
  # transform events
  # most production bugs live here
}

output {
  # send events somewhere
}

Inputs collect. Outputs send. Filters spend CPU.

Demo 2A: Parse with dissect

filter {
  dissect {
    mapping => {
      "message" => "%{log_ts} %{service} %{level} user=%{user} action=%{action} status=%{status} latency_ms=%{latency_ms}"
    }
  }

  mutate {
    convert => {
      "status" => "integer"
      "latency_ms" => "integer"
    }
  }
}
./commands/02-dissect.sh
Run pipelines/02-dissect.conf. Point out typed fields.

Demo 2B: Parse with grok

filter {
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:log_ts} %{DATA:service} %{LOGLEVEL:level} user=%{DATA:user} action=%{DATA:action} status=%{NUMBER:status:int} latency_ms=%{NUMBER:latency_ms:int}"
    }
  }
}
./commands/03-grok.sh

Both work. Which one do you want at 50k events/sec?

Benchmark Gotchas

  • sincedb can make Logstash process zero events.
  • Use separate --path.data for each run.
  • Avoid rubydebug for benchmarks. Console output dominates.
  • Elasticsearch output measures parser + network + bulk indexing.
  • JVM warmup makes first runs noisy.

Mastery Checklist

  • Can you name the most expensive filter?
  • Can malformed events be quarantined?
  • Can you measure events/sec per pipeline?
  • Can you explain queue growth?
  • Can your schema survive arbitrary input?
Do not ask “Can Logstash do this?” Ask “Should Logstash do this on every event, forever?”
