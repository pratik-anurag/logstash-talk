Logstash: The Cost of Every Event

From simple pipelines to production bottlenecks

Pratik

Open with a show of hands: who has tried to fix a slow Logstash pipeline by increasing heap, workers, or batch size first? Set the frame: this talk is not anti-Logstash. It is about treating every filter as code that runs for every event.

Delayed Logs

Logs are delayed by 10 minutes. What knob do you turn first?
  • Increase Logstash heap?
  • Add pipeline workers?
  • Increase batch size?
  • Blame Elasticsearch?

Usually: measure the pipeline before turning knobs.

Let the room vote before revealing the last line. The point is not that these knobs are bad. The point is that they are second-order until we know where time is being spent. Slow pipelines usually come from filter cost, output pressure, queue behavior, or schema/indexing effects downstream.

Working Principle

Logstash filters are production compute, not glue code.

Usually cheaper

Normalize at source, parse repeated shapes, keep mutations shallow.

Cost grows quickly

Regex-heavy parsing, Ruby filters, high-cardinality field names, external lookups.

Emphasize that Logstash is excellent at controlled ingestion and transformation. The risk is putting unbounded or hard-to-predict work into the hot path and then scaling traffic into it. Dynamic field names are especially dangerous downstream because they can create mapping/cardinality pressure even when Logstash itself is still moving events.

Mental Model

input {
  # collect events
}

filter {
  # transform events
  # most production bugs live here
}

output {
  # send events somewhere
}

Inputs collect. Outputs send. Filters spend CPU.

Keep the model simple: inputs decide what becomes an event, filters spend CPU per event, outputs introduce downstream coupling. Most examples in the talk intentionally isolate filters so parser cost is easier to see.

Stable Log Shape

2026-05-24T10:15:30Z api-01 INFO user=pratik action=login status=200 latency_ms=34
Do we need regex for this?

No, if the format is deterministic.

This line has fixed separators and named key-value fragments. That makes it a good candidate for Dissect or a source-side structured format.

Demo 2A: Parse with Dissect

filter {
  dissect {
    mapping => {
      "message" => "%{log_ts} %{service} %{level} user=%{user} action=%{action} status=%{status} latency_ms=%{latency_ms}"
    }
  }

  mutate {
    convert => {
      "status" => "integer"
      "latency_ms" => "integer"
    }
  }
}
./commands/02-dissect.sh
Run the command and point out that `status` and `latency_ms` are converted after extraction. Mention the key tradeoff: Dissect is fast for reliably repeated text and brittle when the shape varies.

Demo 2B: Parse with Grok

filter {
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:log_ts} %{DATA:service} %{LOGLEVEL:level} user=%{DATA:user} action=%{DATA:action} status=%{NUMBER:status:int} latency_ms=%{NUMBER:latency_ms:int}"
    }
  }
}
./commands/03-grok.sh

Both work. Which one should sit on the hot path?

Grok is the right tool when log shape varies or you need existing pattern libraries. The point is to choose it deliberately, test it with representative volume, and avoid broad patterns where narrow ones would work.

Fill this slide only after running `./commands/05-benchmark.sh`.

Real time is the wall-clock time from the benchmark output. Events/sec is `100000 / real_time_seconds`. For example, if real time is 8.0 seconds, throughput is 12,500 events/sec.

The expected direction is dissect fastest, normal grok slower, expensive grok slowest. If the order is different, use that as a debugging prompt: JVM warmup, file state, output pressure, Docker resources, or a benchmark mistake may be dominating the parser cost.

Make clear that production sizing needs representative data, outputs, pipeline workers, heap, downstream behavior, and repeated runs.

Benchmark Gotchas

State sincedb can process zero events

Use a fresh --path.data or disable/recreate file input state for repeatable runs.

Output Console output dominates timing

Avoid rubydebug and noisy stdout when measuring parser cost.

Scope Elasticsearch changes the benchmark

It adds network, bulk indexing, mappings, refresh behavior, and rejection pressure.

Runtime First runs are noisy

JVM warmup, filesystem cache, CPU scaling, and background processes move the number.

Data Synthetic logs can lie

Use representative message sizes, failure rates, optional fields, and worst-case patterns.

Method One run is not a result

Run several times, discard warmup, compare medians, and change one variable at a time.

Decide what you are measuring before trusting the number.

This section does make sense because it protects the audience from false confidence. The benchmark is useful for direction, but only if the measured system is actually the parser or filter cost we intended to isolate.

State: file input state can make a second run process zero events. Use a separate `--path.data`, clear sincedb, or generate a fresh input path when repeating file-input benchmarks.

Output: `rubydebug` and console output can dominate runtime. For parser comparisons, output to `null`, a controlled file, or another minimal sink where possible.

Scope: Elasticsearch output is a different benchmark. It measures parser cost plus network latency, bulk indexing, index template behavior, refresh settings, shard pressure, and possible rejection retries.

Runtime: JVM warmup, filesystem cache, CPU frequency scaling, Docker resource limits, and background laptop work can shift results. Repeat runs and avoid trusting the first number.

Data: synthetic logs often miss optional fields, malformed lines, unusually long payloads, and regex worst cases. Use representative samples and include known bad lines.

Method: change one variable at a time. Keep input, output, worker count, batch size, heap, and pipeline settings fixed when comparing dissect versus grok or one grok pattern versus another.

A parser failure is manageable. An unobserved parser failure is data loss in slow motion.

Grok adds `_grokparsefailure` by default when no pattern matches. The operational move is to route, count, and sample failures instead of letting malformed events blend into normal data.

Backpressure Signals

queue.type: persisted
path.queue: /var/lib/logstash/queue
queue.max_bytes: 1gb
./commands/08-node-stats.sh
  • events.in, events.filtered, events.out
  • duration_in_millis
  • queue_push_duration_in_millis

Persistent queues buffer pressure. When they fill, inputs are back-pressured.

Persistent queues are useful for resiliency and temporary downstream stalls, but they are bounded. If the queue keeps growing, the system is spending stored capacity. Once the limit is reached, Logstash applies back pressure to inputs that support it.

Hot Path Review

grok {
  match => {
    "message" => "%{GREEDYDATA:raw_prefix} user=%{DATA:user} action=%{DATA:action} payload=%{GREEDYDATA:payload}"
  }
}

ruby {
  code => 'event.set("payload_length", event.get("payload").length)'
}

mutate {
  add_field => { "tenant_%{user}" => "active" }
}
./commands/09-final-debugging.sh

Review cost, cardinality, and failure behavior before this ships.

Walk through three review points: `GREEDYDATA` at the front can do more work than needed, Ruby code must be safe and cheap under concurrency, and dynamic field names like `tenant_%{user}` can create mapping/cardinality pressure downstream. Ask what should be moved to the source, narrowed, or quarantined.

Operating Checklist

01 Most expensive filter?

Check plugin duration. Replace broad grok with dissect/json where shape is stable. Move expensive enrichment upstream.

02 Malformed events?

Tag failures. Send bad events to quarantine. Sample payloads and alert on failure-rate changes.

03 Events/sec?

Track in/out rates per pipeline. Benchmark with known samples. Compare before and after each filter change.

04 Queue growth?

Check output latency, bulk rejections, and queue push time. Scale or fix the slow stage, not every knob.

05 Schema safety?

Allowlist fields. Nest unknown payloads. Disable dynamic field creation where mapping pressure is risky.

Do not ask "Can Logstash do this?" Ask "Should this run on every event?"

Close by shifting the habit from capability to cost.

Can you name the most expensive filter?
Remediation: use the node stats API and pipeline stats, especially `events.duration_in_millis`, plugin-level duration, worker utilization, and repeated benchmark runs with representative data. Replace broad grok patterns with dissect or JSON when the shape is stable. Move costly enrichment, Ruby code, or external lookups out of the hot path where possible.

Can malformed events be quarantined?
Remediation: tag `_grokparsefailure`, JSON parse failures, missing required fields, and type conversion failures. Route them to a dead-letter index, file, or separate pipeline. Keep a small sample for debugging and alert on failure-rate changes.

Can you measure events/sec per pipeline?
Remediation: expose `events.in`, `events.filtered`, and `events.out` rate for each pipeline. Run a known sample through the pipeline before and after filter changes. Node-wide throughput is useful, but it can hide one slow pipeline behind other healthy ones.

Can you explain queue growth?
Remediation: treat queue growth as events entering faster than they leave. Check output latency, Elasticsearch bulk rejections, network delay, persistent queue settings, queue push duration, and filter duration before changing heap or worker counts.

Can your schema survive arbitrary input?
Remediation: block dynamic field names, use field allowlists or ECS-style mappings, nest risky payloads under controlled keys, and use index templates to prevent mapping explosions.

The safest Logstash pipelines are explicit about parser choice, failure routing, throughput measurement, queue behavior, and schema boundaries.
